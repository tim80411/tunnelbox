# 本地靜態網頁管理工具 — Spec

## 專案概覽

- **專案名稱**：本地靜態網頁管理桌面應用程式
- **階段**：MVP 第一版
- **撰寫日期**：2026-03-18
- **修訂版本**：v2（品質審查後修正）

### 受益者

| 角色 | 描述 |
|------|------|
| **開發者** | 使用桌面應用建立、管理本地靜態網頁進行開發 |
| **測試環境管理者** | 需要大量建立本地測試環境提供其他人員測試 |

### 期望成果

開發者能透過一個桌面應用，在 30 秒內建立一個帶有 Hot Reload 的本地靜態網頁，並且可以簡單地進行大量網站的管理。

### No-gos

- 不做 Cloudflare 串接（後續 Release 3）
- 不做內建網頁編輯器（使用者用自己的編輯器）
- 不做多人協作功能
- 不做雲端同步或備份

## 修訂紀錄

### v1 → v2 變更摘要

- **合併 Enabler**：解決 Enabler 之間的依賴鏈違反 INVEST Independent 原則的問題
  - 舊 Story 1 + 舊 Story 11 → 新 Story 1（應用程式框架與生命週期管理）
  - 舊 Story 5 + 舊 Story 6 → 新 Story 5（啟動本地靜態伺服器，含自動 Port 分配）
  - 舊 Story 8 + 舊 Story 15 → 新 Story 7（監聽檔案變更，含防抖動機制）
- **補充 Error Path**：Story 1、5、6 新增錯誤情境
- **修正 GWT 格式**：Story 17（刪除確認對話框）的 When 拆分為 Given + When
- **量化 AC**：Story 7 的「合理延遲」改為「500ms 內」
- **修正標題**：Story 3 的「按鈕」改為更中性的用詞
- **重新編號**：全部 Stories 從 1 重新連續編號（1 Spike + 18 Stories = 19 個）

### [Story 1] 應用程式框架與生命週期管理 (Enabler)

**h2. 描述**

作為開發團隊，我們需要建立桌面應用程式的基礎框架、主畫面，以及應用程式啟動與關閉時的資源管理機制，使所有後續功能有統一的承載介面，並確保系統資源不被遺留佔用。

**h2. 價值陳述**

此項變更賦能了所有 User Story（Story 2–9），提供應用程式的基礎運行環境、主畫面框架，以及完整的生命週期管理（包含關閉時自動停止所有伺服器、釋放 Port）。

**h2. 驗收標準**

情境 1: 正常啟動應用程式（Happy Path）
GIVEN 使用者已安裝應用程式
WHEN 使用者開啟應用程式
THEN 應用程式在 3 秒內顯示主畫面
  AND 主畫面包含網頁清單區域與操作區域

情境 2: 應用程式視窗操作（Happy Path）
GIVEN 應用程式已啟動
WHEN 使用者調整視窗大小
THEN 畫面內容自適應新的視窗尺寸

情境 3: 正常關閉時清理資源（Happy Path）
GIVEN 有 3 個網頁的本地伺服器正在運行
WHEN 使用者關閉應用程式
THEN 所有運行中的伺服器停止
  AND 對應的 Port 被釋放

情境 4: 強制關閉（Edge Case）
GIVEN 有網頁伺服器正在運行
WHEN 應用程式被作業系統強制終止
THEN 伺服器進程隨之終止（不殘留孤兒進程）

情境 5: 啟動失敗（Error Path）
GIVEN 使用者已安裝應用程式
WHEN 應用程式啟動過程中發生非預期錯誤
THEN 應用程式顯示錯誤訊息而非無回應地掛起

**h2. 不含（Out of Scope）**

- 不含系統托盤功能
- 不含多視窗支援
- 不含系統托盤常駐功能

**h2. 依賴**

- [Spike] 桌面應用框架選型評估 — 需先確定技術框架

**h2. 優先序**

P0 — 所有功能的基礎承載框架與資源管理，被所有 User Story 依賴

主畫面中顯示網頁清單，每個項目至少顯示網頁名稱與資料夾路徑。

THEN 清單顯示所有已建立的網頁

GIVEN 使用者已建立超過 20 個本地網頁
WHEN 使用者瀏覽清單
THEN 清單可捲動瀏覽
  AND 不影響操作流暢度

**h2. 依賴**

- Story 1（應用程式框架與生命週期管理）— 需要主畫面框架

**h2. 描述**

身為開發者，我需要輸入網頁名稱並指定本地資料夾路徑，以便應用程式知道要為哪個資料夾建立靜態網頁服務。

### [Story 5] 啟動本地靜態伺服器（含自動 Port 分配）(Enabler)

**h2. 描述**

作為開發團隊，我們需要實作本地靜態檔案伺服器功能（包含自動 Port 分配），使使用者建立的每個網頁都能透過本地 HTTP 存取，且多個網頁的伺服器能同時運行不衝突。

**h2. 價值陳述**

此項變更賦能了 Story 6（在瀏覽器開啟網頁）和 Story 8（Hot Reload），使靜態檔案能透過 HTTP 協議提供服務，並確保多站點可並行運行。

**h2. 驗收標準**

情境 1: 正常啟動伺服器（Happy Path）
GIVEN 使用者已指定一個包含靜態檔案的資料夾
WHEN 系統啟動該資料夾的本地伺服器
THEN 伺服器在自動分配的 Port 上提供該資料夾下的靜態檔案
  AND 該 Port 在指定範圍內（例如 3000–9000）
  AND 透過瀏覽器存取 `localhost:{port}` 可看到資料夾中的 index.html 內容

情境 2: 多個伺服器同時運行（Happy Path）
GIVEN 已有 3 個網頁的伺服器正在運行
WHEN 使用者新增第 4 個網頁
THEN 第 4 個網頁取得不同於前 3 個的 Port
  AND 4 個伺服器同時正常運行

情境 3: 資料夾中無 index.html（Edge Case）
GIVEN 使用者指定的資料夾不包含 index.html
WHEN 系統啟動該資料夾的本地伺服器
THEN 伺服器正常啟動
  AND 透過瀏覽器存取時顯示資料夾內的檔案目錄列表

情境 4: 伺服器啟動失敗（Error Path）
GIVEN 使用者指定的資料夾權限不足或無法讀取
WHEN 系統嘗試啟動本地伺服器
THEN 應用程式顯示明確的錯誤訊息
  AND 不影響其他已運行的伺服器

情境 5: 無可用 Port（Error Path）
GIVEN 指定 Port 範圍內所有 Port 皆被佔用
WHEN 系統嘗試啟動伺服器
THEN 應用程式顯示「無可用 Port」的錯誤訊息
  AND 不影響其他已運行的伺服器

**h2. 不含（Out of Scope）**

- 不含 HTTPS 支援
- 不含自訂 MIME type 設定
- 不含 API proxy 功能
- 不含使用者自訂 Port 功能

**h2. 依賴**

- 無

**h2. 優先序**

P0 — 被 Story 6、8 依賴，是網頁預覽與 Hot Reload 功能的基礎

**Labels**: `P0`, `enabler`

### [Story 7] 監聽檔案變更（含防抖動機制）(Enabler)

**h2. 描述**

作為開發團隊，我們需要實作檔案系統監控功能（含防抖動機制），使系統能偵測使用者對靜態網頁檔案的修改，並在批次變更時避免觸發過多通知。

**h2. 價值陳述**

此項變更賦能了 Story 8（自動重新載入瀏覽器），是 Hot Reload 功能的技術基礎。防抖動機制確保批次操作（如 git checkout、npm build）不會觸發大量無意義的重新載入。

**h2. 驗收標準**

情境 1: 偵測檔案修改（Happy Path）
GIVEN 系統正在監聽某個資料夾
WHEN 使用者修改該資料夾下的任意檔案（HTML / CSS / JS / 圖片等）
THEN 系統在 1 秒內偵測到檔案變更事件

情境 2: 偵測新增與刪除檔案（Happy Path）
GIVEN 系統正在監聽某個資料夾
WHEN 使用者在該資料夾下新增或刪除檔案
THEN 系統偵測到對應的檔案變更事件

情境 3: 子資料夾的變更（Edge Case）
GIVEN 系統正在監聽某個資料夾
WHEN 使用者修改該資料夾的子資料夾中的檔案
THEN 系統同樣偵測到檔案變更事件

情境 4: 批次檔案變更的防抖動（Happy Path）
GIVEN 系統正在監聽某個資料夾
WHEN 短時間內有多個檔案同時被修改（如 git checkout）
THEN 系統將多次變更合併為一次通知
  AND 通知發生在最後一次檔案變更後 500ms 內

情境 5: 監控中的資料夾被刪除（Error Path）
GIVEN 系統正在監聽某個資料夾
WHEN 該資料夾被外部操作刪除或移動
THEN 系統停止對該資料夾的監聽
  AND 不產生崩潰錯誤

**h2. 不含（Out of Scope）**

- 不含忽略特定檔案/資料夾的篩選規則（如 node_modules）
- 不含使用者自訂防抖動延遲時間

**h2. 依賴**

- 無

**h2. 優先序**

P0 — 被 Story 8 依賴，是 Hot Reload 的核心基礎

**Labels**: `P0`, `enabler`

### [Story 8] 檔案變更時自動重新載入瀏覽器頁面

**h2. 描述**

身為開發者，我需要修改本地檔案後瀏覽器自動更新，以便即時看到修改效果而不需手動重新整理。

**h2. 範圍**

當監測到檔案變更時，通知瀏覽器重新載入頁面。

**h2. 驗收標準**

情境 1: 修改檔案後瀏覽器自動更新（Happy Path）

WHEN 使用者修改該網頁資料夾下的檔案
THEN 瀏覽器頁面在 2 秒內自動重新載入

情境 2: 多個瀏覽器分頁同時開啟同一網頁（Edge Case）
GIVEN 使用者在多個瀏覽器分頁中開啟了同一個本地網頁
WHEN 使用者修改該網頁的檔案
THEN 所有開啟的分頁都自動重新載入

**h2. 依賴**

- Story 5（啟動本地靜態伺服器）— 需要運行中的伺服器

### [Story 9] 從清單中移除網頁並停止對應伺服器

**h2. 描述**

身為開發者，我需要能從清單中移除不再需要的網頁，以便保持管理清單的整潔。

**h2. 範圍**

從清單中移除網頁記錄，同時停止對應的本地伺服器。不刪除使用者的本地檔案。

**h2. 驗收標準**

情境 1: 正常移除網頁（Happy Path）
GIVEN 清單中有一個已建立的網頁
  AND 該網頁的伺服器正在運行
WHEN 使用者對該網頁執行移除操作
THEN 該網頁從清單中消失
  AND 對應的本地伺服器停止運行
  AND 使用者的本地檔案不被刪除

情境 2: 移除伺服器已停止的網頁（Edge Case）
GIVEN 清單中有一個網頁但其伺服器已停止
WHEN 使用者對該網頁執行移除操作
THEN 該網頁從清單中消失

### [Story 18] 下次啟動時恢復上次的網頁清單設定 (Enabler)

**h2. 描述**

作為開發團隊，我們需要將網頁清單持久化儲存，使使用者重新開啟應用程式後不需重新設定所有網頁。

**h2. 價值陳述**

此項變更使 Story 2（顯示網頁清單）在重新開啟應用後仍能顯示先前建立的網頁，大幅提升大量網站管理的實用性。

**h2. 驗收標準**

情境 1: 重新開啟後恢復清單（Happy Path）
GIVEN 使用者已建立 5 個本地網頁
  AND 使用者關閉應用程式
WHEN 使用者重新開啟應用程式
THEN 主畫面顯示先前建立的 5 個網頁
  AND 各網頁的伺服器自動重新啟動

情境 2: 持久化資料損壞（Error Path）
GIVEN 持久化儲存的資料損壞或格式不符
WHEN 使用者開啟應用程式
THEN 應用程式正常啟動並顯示空清單
  AND 不產生崩潰錯誤

**h2. 不含（Out of Scope）**

- 不含跨裝置同步
- 不含匯出/匯入設定

**h2. 依賴**

- 無（持久化邏輯可獨立於 UI 框架開發）

**h2. 優先序**

P2 — 持久化是管理大量網站的必要功能，提升實用性

**Labels**: `P2`, `enabler`
